Дам опинионейтед план — с конкретными рекомендациями, а не «рассмотрите варианты». На 2 недели расплывчатость убийственна.

## Ключевой принцип

Оркестрация — это то, за что даётся хакатон. Написание кода агентом — уже решённая задача. Поэтому **берите готовый coding-агент** (Aider, Claude Code SDK, или OpenCode) как «кодовый движок» и стройте свою оркестрацию поверх него. Попытка с нуля написать агента, который редактирует файлы, управляет контекстом и применяет диффы, съест всю неделю и не даст ничего уникального.

## Архитектура

Оркестратор — это конечный автомат с явными стадиями. Каждая задача проходит по пайплайну, её состояние персистится между стадиями, чтобы при падении можно было возобновить с нужного места.

**Стадии пайплайна:**

1. **Fetch** — забирает задачу из источника (GitHub Issue с лейблом `agent-task` — самый простой вариант, не требует инфры).
2. **Context** — читает репозиторий, собирает релевантные файлы (по путям из задачи, по grep, или через семантический поиск — от простого к сложному).
3. **Plan** — LLM получает задачу + контекст, возвращает план: какие файлы трогать, что в них менять, какие тесты должны пройти. Структурированный JSON.
4. **Implement** — план + контекст отдаются coding-агенту, который вносит изменения в рабочую копию (git worktree).
5. **Verify** — прогоняет линтер, тесты, билд в песочнице (Docker или просто subprocess в worktree).
6. **Retry loop** — если verify упал, лог ошибок + текущий дифф возвращаются в Implement с пометкой «вот что ты сделал, вот что сломалось». Лимит 3 попытки.
7. **PR** — коммит, пуш, открытие PR через GitHub API.
8. **Monitor** (опционально) — следит за CI на PR, комментирует, уходит на доработку.

**Состояние и персистентность.** SQLite с таблицей `tasks(id, status, current_stage, attempts, logs_json, pr_url, ...)`. Каждая стадия — отдельная функция, принимает состояние, возвращает обновлённое. Это даёт возможность ретраить с конкретной стадии, отлаживать на реальных данных и даёт основу для observability.

**Изоляция.** Для каждой задачи — отдельный git worktree в `/tmp/agent-workdirs/<task-id>/`. Это позволяет параллелить задачи и не ломать основной чекаут. Для безопасности верификации — Docker-контейнер с прокинутым worktree.

## Стек — конкретные выборы

**Язык:** Python. Единственный разумный выбор из-за экосистемы агентов и LLM-библиотек.

**Источник задач:** GitHub Issues с лейблом. Poll каждые 30 секунд — проще вебхуков, не нужна публичная ручка. Если кто-то в команде фанат Linear/Jira — их API тоже тривиальны, но GitHub уже где лежит код.

**Оркестрация.** Тут выбор между «написать самим» и LangGraph. Для команды новичков в агентах LangGraph даёт приятные абстракции (узлы, рёбра, conditional edges, состояние). Для опытной команды — проще написать свой конечный автомат на 200 строк, будет чище и без магии. По моему опыту, своё обычно быстрее для такого размера.

**Coding-агент.** Aider — самый зрелый и простой в интеграции, есть Python API. Claude Code SDK — тоже хороший вариант, особенно если кто-то в команде с ним уже знаком. Оба справятся. Не пытайтесь сами реализовывать редактирование файлов через tool use — потратите 3 дня.

**LLM.** Для планирования — модель посильнее (Claude Sonnet 4.5 / GPT-5 / Gemini 2.5 Pro). Для имплементации — та же или чуть слабее, но не сильно дешевле: на коде экономия на модели почти всегда выходит боком. Это, кстати, пересекается с опциональным блоком «выбор стратегии от типа задачи» — можно потом попробовать.

**GitHub integration.** PyGithub для API + `gh` CLI или `gitpython` для локальных операций. Оба работают, не принципиально.

**Observability.** Структурированные логи (structlog) + Langfuse для LLM-вызовов (бесплатный self-hosted, ставится за 10 минут через docker-compose). Langfuse даст view на все промпты, токены, стоимость — это сильно поможет и на демо смотрится хорошо.

**Sandbox.** Docker-контейнер с предустановленным окружением репозитория, примонтированный worktree. Если хочется проще — просто subprocess с таймаутом, без изоляции, работает на доверенном репо.

## Таймлайн по дням

**День 1–2: Skeleton без интеллекта.**
Весь пайплайн end-to-end, но с заглушками. Fetch читает GitHub Issue. «Plan» и «Implement» возвращают хардкод — например, добавляют строчку в README. Verify запускает `echo "ok"`. PR реально создаётся. Цель — чтобы по команде `run-orchestrator` задача из Issue превращалась в PR с изменённой строкой. Без LLM вообще.

Это самый важный этап. Он даёт скелет, на который дальше навешивается всё остальное, и обнаруживает интеграционные проблемы (права на репо, API-токены, структура worktree) пока они дешёвые.

**День 3–4: Подключаем LLM.**
Стадии Plan и Implement становятся реальными. Берёте Aider как кодовый движок. На этом этапе запускаете на игрушечных задачах типа «добавь функцию, которая складывает два числа, и тест к ней». Не надейтесь, что сразу заработает на сложном — цель увидеть, что весь цикл проходит.

**День 5–6: Verify и retry loop.**
Добавляете реальную верификацию — pytest, ruff, mypy (что есть в выбранном репо). Делаете честный фидбэк-цикл: если тест упал, в промпт к Implement летит ошибка + текущий дифф. Калибруете лимит ретраев и что именно передаёте обратно (сырой stderr vs. отформатированный отчёт).

**День 7: КОНТРОЛЬНАЯ ТОЧКА.**
К этому дню end-to-end должен работать на реальной задаче в реальном репозитории. Пусть криво, пусть с 50% успешностью — но работать. Если не работает — всё что после этой точки урезается в пользу доведения MVP.

**День 8–10: Один опциональный блок.**
Выбираете ОДИН опциональный блок (см. ниже). Не два. Реализуете, тестируете, интегрируете.

**День 11–12: Прогон и полировка.**
Создаёте в реальном репозитории 3–5 разных Issue-ов (простые и посложнее), запускаете оркестратор, смотрите что получается. Фиксите самое болезненное. Улучшаете промпты на основе реальных провалов — это почти всегда самый дешёвый апгрейд качества.

**День 13–14: Демо.**
Сценарий демо, запись, возможно веб-дашборд со статусом задач (если успеется — мощно смотрится). Заметки для рассказа: архитектурные решения, что пробовали и не зашло.

## Какой опциональный блок взять

Ранжирую по соотношению «ценность для демо / стоимость»:

**Observability** — дешёво (Langfuse уже будет стоять) и очень хорошо смотрится. Показываете live дашборд на демо — это вау-эффект.

**Agentic code review** — средняя стоимость, высокая ценность. Отдельный агент с промптом «ты ревьюер» смотрит на дифф и либо аппрувит, либо возвращает с комментариями. Замыкает петлю качества и на демо хорошо рассказывается.

**Quality gates** — если в репо уже есть тесты/линтеры, вы фактически это уже сделали в Verify. Минимальная доработка → считается сделанным блоком.

**Декомпозиция** — соблазнительно, но тяжело. Если возьмётесь, готовьтесь что сожрёт все 3 дня и не факт что заработает. Берите, только если команда сильная и готова рискнуть.

**Долгосрочная память** — интересно, но 3 дня мало, чтобы увидеть эффект. Для хакатона слабо.

Моя рекомендация: **Observability + Agentic code review**. Observability идёт почти бесплатно, а review даёт содержательную фичу.

## Чего НЕ делать

Не писать своего агента с tool use для редактирования файлов. Не использовать Temporal или Airflow (overkill). Не пытаться поддержать несколько LLM-провайдеров через абстракцию — выберите одного и живите с ним. Не делать веб-UI до дня 12 — CLI достаточно. Не поддерживать несколько языков/фреймворков в целевом репо — выберите один репо на Python/JS и оптимизируйте под него.

## Что обсудить с командой на техническом созвоне

Главные спорные точки, которые лучше закрыть голосованием а не спором: (1) свой конечный автомат vs LangGraph, (2) Aider vs Claude Code SDK, (3) какой целевой репозиторий, (4) какой LLM-провайдер основной. На каждое — максимум 10 минут.
